# 同步与安全设计

## 数据路径优先级

Woo Todo 默认不使用公网任务同步。安装后不注册 Woo Todo 账号、不创建同步空间，也不向 Worker 或坚果云发送任务；macOS 与 Android 始终先读写各自的本地 SQLite。应用最多每 24 小时请求一次 GitHub 公共 Release API 检查正式版，只传递常规 HTTP 元数据，不包含任务或同步凭据；失败后至少等待 15 分钟才重试。

跨端有两条彼此独立的在线路径：

1. **坚果云自动同步（推荐）**：使用坚果云 WebDAV 保存每个操作的端到端加密对象，不需要自建服务器；
2. **自建 Worker 同步**：用户自行部署 Worker、创建同步空间并完成二维码配对后，客户端上传端到端加密的增量操作。

无论使用哪条路径，网络和远端服务都不是新增、编辑、完成、Pass、删除或排序的前置条件。

## 坚果云 WebDAV 自动同步

坚果云模式使用固定的 `https://dav.jianguoyun.com/dav/` 端点。用户应先安装并登录坚果云官方客户端；网页端只用于账户安全设置和生成应用密码，Woo Todo 内置 WebDAV 客户端直接连接该端点。应用密码只保存在 macOS Keychain 或 Android Keystore 包装的本地凭据中，通过 TLS Basic Auth 访问 WebDAV，不会写入任务数据库、同步对象或日志。推荐先在 Mac 配置；连接成功后二维码默认隐藏，用户显式点击“显示 Android 配置二维码”，再由 Android App 内置扫码器解析并显示确认页。二维码严格包含 `username`、`appPassword`、`vaultId` 和 `vaultKey`，不含固定 WebDAV 端点或 `deviceId`；Android 确认保存时生成本机独立 `deviceId` 并立即同步。该二维码同时属于坚果云 WebDAV 与 Woo Todo 的完整敏感凭据，只能近旁展示，用完立即隐藏，不能上传 Woo Todo 服务、公开渠道或日志，客户端也不得记录完整二维码或配置链接。系统相机打开深链仍作为备用，但 `wootodo://` 自定义 scheme 不能证明处理应用归属，必须确认目标确为 Woo Todo。两端最终使用同一坚果云账号、同步空间名和同步密钥，设备 ID 必须不同；若要分别撤销 WebDAV 权限，可在扫码后替换为 Android 专用应用密码。

每个变更写成一个不可变对象：

```text
v1/<vaultId>/ops/<opId 前两位>/<opId>.json
```

对象正文和任务标题均由 AES-256-GCM 加密。客户端按 `opId` 幂等应用，失败事务会回滚任务、版本水位和已应用标记；tombstone 永久阻止同一实体 ID 复活。WebDAV 模式不使用 Worker 游标，重复扫描是有意设计，方便在没有服务端数据库或 webhook 的情况下收敛。

配置步骤、应用密码入口和通知设置见 [坚果云自动同步与通知](JIANGUOYUN_SYNC.md)。

## 为什么不使用夸克网盘自动同步

夸克 SVIP 提供文件存储权益，不等同于稳定的应用数据 API。让两台设备同时覆盖 SQLite 或整份 JSON，会产生并发覆盖、半写入和无法解释的冲突。

夸克网盘可以作为用户手动保存加密 `.wootodo` 恢复备份的文件介质，但客户端不会监控网盘目录、后台轮询或自动覆盖同一个文件。它不能充当跨端同步服务，恢复文件也不能合并到已有任务库。

## 自建 Worker 在线 Local-first 流程

用户自行启用 Worker 同步后：

1. 用户操作在本地事务中更新任务并写入持久化 outbox；
2. UI 立即读取本地数据库，不等待网络；
3. 同步引擎在平台允许的时机批量加密并上传 outbox；
4. 服务端按 `op_id` 幂等接收，分配单调递增变更序号；
5. 客户端从已确认游标之后拉取其他设备的密文变更；
6. 客户端验证、解密并在本地事务中合并，然后推进游标。

上传成功但响应丢失时，同一 `op_id` 可安全重试。服务端只保存密文和同步所需元数据，不接触任务标题。

### 离线与省电边界

- 已配置在线同步但暂时无网时，本地操作和 outbox 均保留，网络恢复后再发送；
- Android 使用带网络约束和指数退避的 `JobScheduler`，不创建前台常驻服务；
- macOS 只在本地变化、启动、唤醒、网络恢复和低频兜底时请求同步，不高频轮询；
- Keychain/Keystore 暂时不可用时不覆盖同步身份，本地任务仍可编辑，凭据恢复后补录真实变化；
- 两端都没有公网时，各自继续使用本地数据，但无法交换新变化；恢复联网后，已配置的同步会继续处理积压操作。v1 不实现局域网自动发现或蓝牙后台同步。

## 可选在线配对

首台设备创建同步空间并生成 `vault_key`。新增设备配对时：

1. 已绑定设备向服务端创建 10 分钟有效的一次性会话并显示二维码；
2. 新设备上传 X25519 临时公钥；
3. 两端显示相同的六位人工核对码；
4. 旧设备确认后，用协商出的会话密钥加密传递 `vault_key` 和新设备令牌；
5. 新设备把密钥保存到系统安全存储，配对会话立即失效。

二维码不包含长期同步密钥。生产构建只接受双方可访问的 HTTPS Worker；`127.0.0.1` 在手机上表示手机自己，不能用于双机配对。完整操作见 [可选在线配对同步](PAIRING.md)。

配对派生必须在两端逐字一致：

- X25519 计算 32 字节共享秘密；
- HKDF-SHA256 的 salt 为 32 字节 `pairingSecret`，info 为 `woo-todo-pairing-v1|pairingId`；
- 六位码使用 session key 对 `woo-todo-pairing-code-v1|initiatorPublicKey|claimPublicKey` 做 HMAC-SHA256；
- vault key envelope 的 AAD 为 `woo-todo-pair-v1|pairingId|claimedDeviceId`。

固定向量位于 `shared/fixtures/crypto-vectors.json`，Swift、Kotlin 和 Node 必须共同通过。

## 在线密文与服务端边界

- 算法：AES-256-GCM；每个操作使用随机 96 位 nonce；
- AAD：`woo-todo-sync-v1|vaultId|opId|entityId|kind|lamport|deviceId`；
- ciphertext：JSON payload 密文与 128 位认证标签，使用无填充 Base64URL；
- 服务端可见同步空间、设备、实体 ID、操作 ID、逻辑时钟、密文大小和变更序号；
- 服务端不得记录请求正文、原始设备令牌、二维码内容或解密后的诊断信息；
- 设备令牌只保存哈希，`vault_key` 只进入 Keychain/Keystore 和用户主动导出的加密恢复包。

## 恢复与撤销

- 加密恢复备份只允许导入空白安装，不与已有任务库合并；
- 替换原设备时可恢复原逻辑设备身份，新增并存设备应配置同一个坚果云同步空间，或使用 Worker 二维码配对；
- 设备撤销后令牌立即失效，但无法擦除该设备此前下载的本地副本；
- 全部设备、恢复文件和口令同时丢失时，服务端密文不可恢复；
- 当前客户端不会在服务端数据丢失后自动重传全库，应保留至少两份独立的加密恢复备份。

## 运维边界

- Worker 对每个请求的变更数量、密文总量、设备数和单空间操作数设置硬限制；
- 设备令牌按 IP 与 vault 做速率限制；
- 当前不做服务端明文快照、日志 compaction 或自动灾备；
- Cloudflare 免费额度和中国大陆网络连通性属于外部条件，部署当天必须自行验证；
- Worker 未部署或不可达不影响本地离线使用；未自行部署 Worker 时可选择坚果云自动同步。
